# v0.9.4 Unified Live Room PTY Dogfood

This document records the release-blocking dogfood evidence for issue `#306`.
The purpose is to prove the unified room path with a real terminal user flow,
not only renderer helpers or syntax-level tests.

## Scenario

Command:

```bash
python3 scripts/dogfood-v09-user-cases.py
```

The script now includes a dedicated v0.9.4 live-room PTY scenario:

```text
cr console --live-room
```

It runs inside a 160x48 PTY from a freshly initialized temporary project and
types the same way a user would type:

```text
validate unified room from real pty
@reviewer check explicit routing
/journal reviewer
/exit
```

## What The Scenario Proves

| Case | Evidence |
| --- | --- |
| Non-default unified room entrypoint | `cr console --live-room` opens in a real PTY and exits with `/exit`. |
| User room remains primary | The room renders `CoreRoom Workspace` and `Ask @host` while accepting typed user text in the same terminal surface. |
| Bare user text routes to host | The PTY output includes `validate unified room from real pty` and `@host received the request`. |
| Explicit role routing remains available | The PTY output includes `@reviewer check explicit routing`, reviewer lane activity, and explicit `@role` routing status. |
| Runtime-only slash command gap is explicit | `/journal reviewer` produces the clear `not yet available in cr console --live-room; use cr start` guidance. |
| Dashboard panels stay derived | The scenario verifies `Project`, `Control Rail`, and structured fact panes are present, but does not treat rendered panels as completion authority. |
| No old-REPL fallthrough | The output must not contain `CoreRoom console closed; starting REPL`. |

## Permission Prompt Status

The real engine permission prompt path is intentionally not claimed as complete
in v0.9.4. The live-room bridge added in `#305` routes composer input and stages
user-visible status inside the unified room, but it does not yet spawn role
engines or replace the mature REPL permission socket.

That means `#306` proves the real PTY entrypoint, composer, host routing,
explicit role routing, visible unsupported-command guidance, and safe exit. It
does **not** make the unified room the default `cr` path.

Update for `#313` / v0.9.5: the default-entrypoint decision changed after real
user testing showed the dashboard-to-REPL split was the wrong product shape.
Plain `cr` now opens the unified live room by default. `cr start` remains the
explicit legacy/direct REPL escape hatch while deeper role-execution parity is
tracked separately.

Update for `#315` / v0.9.6: the default live room no longer projects old
`.coreroom/messages.jsonl` history into the main surface. The center panel is
the current-session `CoreRoom Workspace`; read-only `cr console` keeps
transcript projection for inspection and recovery.
