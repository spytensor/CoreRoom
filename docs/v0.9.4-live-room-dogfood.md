# v0.9.4 Live Room Dogfood Archive

This document is historical evidence for issue `#306`. The staged live-room
preview it described was removed by `#330` because the bridge mutated an
in-memory snapshot from the input loop instead of projecting real runtime
events.

## Current Status

`cr console --live-room` no longer opens the staged v0.9.4 preview. After
`#331` and `#332`, it opens the executable full-screen TUI room backed by the
same role runtime used by `cr start`.

This archive must not be used as acceptance evidence for the current runtime
room. Current evidence lives in `docs/v0.9-real-user-dogfood.md` and the
`scripts/dogfood-v09-user-cases.py` run from `#332`.

## Current Dogfood Harness

The v0.9 dogfood script now checks behavior, not preview routing text:

- plain `cr` opens the executable TUI room in a real PTY
- a gated fake engine streams chunks without network/API usage
- the role lane transitions through idle/working/idle
- the permission overlay fires through the mature permission bridge path
- `.coreroom/messages.jsonl` gains durable turn and permission events
- `cr console --live-room` opens the executable TUI room explicitly
- `cr console` and `cr console --snapshot` remain read-only dashboard paths

## Historical Scope

The removed scenario used to open:

```text
cr console --live-room
```

inside a 160x48 PTY from a freshly initialized temporary project, then typed
bare user text, an explicit `@reviewer` request, `/journal reviewer`, and
`/exit`. That evidence only proved the staged bridge rendered input and status
inside a terminal. It did not prove role-engine execution, streaming runtime
events, durable turn state, or permission prompt parity.

## Rollback

Reverting to the staged bridge would require restoring the deleted
`src/console_room.rs` path, the removed preview tests, and the old dogfood
assertions. That would be a product regression unless the executable TUI room
also had to be backed out.
