# v0.10 design — chat stream and right rail dashboard are independent surfaces

- **Status:** accepted
- **Filed:** 2026-05-26
- **Closes part of:** #377 (umbrella), #378 (this ADR)
- **Touches:** Live room IA. Supersedes the v0.9.x model where the right rail and the chat carried overlapping role-activity information.

## Why this ADR exists

Dogfooding v0.9.15 surfaced a modeling mistake. The right rail's `Status` card and the main Room both describe *who is on the team*, *who is working*, *what tool they are calling*, and *what just happened* — through two different visual languages, simultaneously. Two surfaces, one truth, no clear owner. The rail also mixes stable team roster with transient per-frame tool calls in the same tree, so neither layer reads cleanly.

This ADR locks the new model and its vocabulary so the v0.10 child issues (#379–#386) cannot diverge. No runtime change in this PR — it ships the document.

## Decision

The chat stream and the right rail are **independent surfaces with non-overlapping responsibilities**:

- **Chat stream (main Room)** owns *what the team is doing right now*. Each spawned sub-agent appears inline as a working card with a live tool-call stream, collapses to a one-line done marker on completion, and emits its report as a regular chat message. Causality is expressed through visual adjacency in the stream, not through a tree.
- **Right rail dashboard** owns *project / work-item state that outlives any single turn*: Work counts, Cards, Evidence validation, room-level blockers. It does **not** track who is currently typing, what tool is running, or who spawned whom.

When the two surfaces disagree, the chat stream is the source of truth for live activity; the rail is the source of truth for ambient project state.

## Locked vocabulary

Downstream issues use these terms verbatim.

- **Chat stream** — the main Room timeline. The only surface that depicts active sub-agent work.
- **Right rail dashboard** — ambient project state (Work, Cards, Evidence, room-level blockers). Does not depict per-turn sub-agent activity.
- **Sub-agent lifecycle** — `Spawning → Working → Done → Reported`. There is no separate `Failed` state; an interrupted or failed sub-agent reaches `Done` with an annotation (`⨯ interrupted`, `⨯ failed`).
  - `Spawning`: not yet rendered as a card; only visible as a hint inside the spawner's stream output.
  - `Working`: inline working card with title row, live tool-call stream, footer hotkey hints.
  - `Done`: card collapses to a single line `@role ✓ done · {elapsed} · {N steps} · [e]xpand log`.
  - `Reported`: agent emits a regular chat message at its own timestamp. The collapsed line stays in scrollback as a header for that report.
- **Per-card hotkeys** — `[e]xpand` (expand a collapsed-done log), `[i]nterrupt` (cancel one sub-agent without halting the room), `[f]ocus` (zoom one card to full-height, dim others). Hotkeys are scoped to the card that currently has keyboard focus and only fire when a card is focused; otherwise they fall through to the composer as ordinary text input.
- **Footer narration line** — a single line above the composer that names the roles being waited on when sub-agents are running. When nothing is in flight: the line is hidden (no empty row reserved).

## Resolved open questions

These are the levers downstream issues will pull. Each is decided here and is not re-litigated downstream.

1. **Working card stickiness when scrolling.** Cards do **not** pin. They scroll with history. Recovery path is `[f]ocus` (jump back to a working card) and `End` (return to live tail). Rationale: pinning creates a parallel scroll surface that fights normal scrollback; we already have a `↓ N new` follow-back indicator from v0.9.16 #371 that covers the "I have unread work below" case.

2. **Collapsed-done log expansion.** Inline expansion. The chat row grows in place; no side detail panel. Rationale: keeps the chat as the single source of truth, consistent with this ADR's thesis. Side panels would re-introduce a second surface that needs its own scrollback model.

3. **Sub-agent failure modeling.** Lifecycle has **no separate `Failed` state**. A sub-agent that errors out or is interrupted via `[i]nterrupt` (#385) transitions `Working → Done` with a visible annotation — `@role ⨯ interrupted` or `@role ⨯ failed` — that distinguishes the visual from a clean `✓ done`. Rationale: keeps the state machine to four states, lets renderers branch on annotation rather than on extra states.

4. **Footer wording.** The line lists roles only — `2 roles still working · @security @backend`. It does **not** promise "chat resumes when they report", because the production room does not actually gate input on sub-agent activity in v0.10. If we add that gating later, the line gets a suffix; we do not advertise behavior we do not ship. (Closes the wording question raised in #382.)

5. **Right rail scope after the move.** **Slim status strip**, not kill, not on-demand. The rail keeps a single `Status` card, max ~6 lines, with the fields that legitimately outlive a single turn: Work `active: N / cards: M`, Evidence `validation: <state>`, Blockers `state: ...` (room-level only). The removed fields — `activity: K/N roles seen`, `latest: ...`, `assignee: ...`, the embedded tool-call tree — all move to the chat stream. Rationale: complete kill is too disruptive (users have a Status anchor in their mental model after v0.9.13–15); on-demand requires teaching a new `/status` command. Slim preserves the anchor while removing the noise.

## Consequences

**Positive**

- Causality reads top-to-bottom in the chat stream. Tree visualization (delegate-tool-call tree in the rail) is gone; visual adjacency carries the same information with less cognitive load.
- Right rail flicker on every tool call disappears, because the rail no longer reads per-spawn lifecycle records (AC enforced in #383).
- Concurrent sub-agents render naturally — each gets its own card, no card competes for rail space.
- Failure modes (interrupt, error) are visually expressed as the same `Done` marker with an `⨯` annotation, so the visual vocabulary stays small.

**Negative**

- Long sub-agent runs push the user's own conversation up. The `↓ N new` follow-back indicator and `End` keystroke (v0.9.16 #371) mitigate, but heavy users will need `[f]ocus` mode (#385) for sustained attention.
- The rail's `Status` card loses information that some users may have relied on (`activity: K/N roles seen`). The "removed fields → new home" table in #383's PR description compensates.
- A user who scrolls back through history sees historical working cards as they appeared at the time, not as collapsed `Done` markers. This is intentional but worth noting — scrollback is a transcript, not a state mirror.

## Out of scope (for the v0.10 cycle)

- Persisting sub-agent transcripts beyond the current room session.
- Mobile / non-terminal renderings.
- Changing the underlying delegation protocol or sub-agent lifecycle on the kernel side beyond what is needed to expose lifecycle state to the renderer.
- Mouse-clicking a card to focus it (keyboard only in #385).
- Gating composer input while sub-agents work (footer wording is non-aspirational per Q4).
