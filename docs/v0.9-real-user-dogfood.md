# v0.9 Real Local User-Case Dogfood

This document records release-blocking dogfood evidence for issue `#291`,
extended by `#306` for the v0.9.4 unified live-room path, by `#313` for the
v0.9.5 default live-room entrypoint, by `#315` for the v0.9.6 default
workspace correction, and by `#317` / `#318` / `#319` for the v0.9.7
truthfulness hotfix. It exists because
v0.9 release readiness must not rely only on compilation, unit tests, syntax
checks, or static renderer assertions.

## Scenario Matrix

| Scenario | Real command path | Evidence |
| --- | --- | --- |
| Build local user binary | `cargo build --locked --quiet` | Local `target/debug/cr` exists and reports the current `cr` version |
| Fresh user project setup | `cr init --project <temp> --yes --preset team` | `.coreroom/` written with team roles, gate templates, shared priors, and lockfile |
| Role inventory | `cr role list --project <temp>` | `@host`, `@engineer`, `@reviewer`, `@sre`, `@security`, `@qa` listed |
| Host authority prompt | `cr prompt show host --project <temp>` | Prompt contains `You are \`@host\`` and highest-authority wording |
| Priors integrity | `cr verify --project <temp>` | Priors lock verifies against generated project files |
| Default executable entry | `cr` inside a PTY from a fresh initialized project | Plain `cr` reaches the executable CoreRoom runtime splash/prompt, exits with `/exit`, and does not render staged live-room preview tokens such as `CoreRoom Workspace`, `Ask @host`, or `received the request` as the default path |
| Full-screen console entry | `cr console` and `cr console --snapshot tests/fixtures/console_snapshot_v08.toml` inside a PTY | Console enters a real terminal from live local project state or a snapshot fixture, renders project/transcript/environment/role/evidence facts, receives `q`, and exits cleanly |
| Unified live room preview alias | `cr console --live-room` inside a PTY from a fresh initialized project | The explicit alias renders workspace and dashboard facts together, shows a visible input cursor, labels bare text and explicit `@reviewer` text as staged preview routes, explains that role turns are not executing there, and `/exit` leaves the room without old-REPL fallthrough |
| README visuals | `make readme-images` | All README PNGs regenerate and pass PNG/file-size checks |

## Harness

The repeatable local harness is:

```bash
python3 scripts/dogfood-v09-user-cases.py
```

The script intentionally runs real commands rather than only calling Rust helper
functions:

- builds the local binary
- creates a temporary user project with a small Rust app
- runs `cr init --yes --preset team`
- inspects generated role inventory and composed host prompt
- verifies `.coreroom/priors.lock`
- enters `cr console` through a Unix PTY at `120x40`
- enters plain `cr` through a Unix PTY at `160x48`, verifies the executable
  runtime prompt, and exits with `/exit`
- enters `cr console --live-room` through a Unix PTY at `160x48` and types
  real composer input
- sends `q` to exercise terminal restoration / exit path
- regenerates README images

## Real Failures Caught During Harness Work

The harness caught two non-syntax issues while being added:

- The first version asserted the simplified phrase `You are @host`; the real
  composed prompt says `You are \`@host\``. The assertion was corrected to match
  the real user-visible prompt.
- The first PTY assertion looked for `Control Rail` at 120 columns. In the real
  terminal path, control sequences and responsive layout make decorative titles
  less stable than the actual user-facing facts. The harness now validates
  durable user facts: project, conversation, environment, roles, evidence, and
  public user/host transcript markers.

These failures are exactly why this gate exists: realistic local paths expose
problems that pure syntax or helper-level tests do not.

## Not Verified

- Real engine turns with Claude Code, Codex, or Gemini are still excluded from
  this gate because they require authenticated CLIs and can burn user/API quota.
- The PTY scenario validates console entry/exit and visible facts from a
  local project, including the v0.9.2 user-first conversation pane and
  host-managed task-card treatment for internal delegation. It does not
  validate live GitHub polling.
- The v0.9.7 live-room PTY scenario validates typed preview input and staged
  routing only through the explicit `cr console --live-room` alias. It does not
  validate authenticated role-engine execution or the mature REPL permission
  prompt socket inside the ratatui room; those remain explicit runtime-parity
  follow-ups before the live room can become the default again.
- The script uses Unix PTY APIs and is intended for macOS/Linux release
  dogfood. Windows terminal dogfood should be added as a separate platform gate
  before claiming native Windows console support.

## Release Rule

Any future release touching the console, host-led control surface, README
visuals, setup/init path, or role prompt composition must run:

```bash
python3 scripts/dogfood-v09-user-cases.py
```

The result must be included in PR evidence. If this script fails, the release is
not ready even if `cargo test`, `clippy`, and CI pass.

## Rollback

If this dogfood gate proves too broad or flaky, revert the PR that added this
document and `scripts/dogfood-v09-user-cases.py`. The change does not alter
runtime behavior, persisted user state, package metadata, or release artifacts.
