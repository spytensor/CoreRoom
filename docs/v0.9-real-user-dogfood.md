# v0.9 Real Local User-Case Dogfood

This document records release-blocking dogfood evidence for issue `#291`,
extended by `#306` for the original v0.9.4 live-room path, by `#313`,
`#315`, `#317`, `#318`, `#319`, and `#330` for the staged-room correction
series, and by `#331` / `#332` for the executable full-screen TUI room.
It exists because v0.9 release readiness must not rely only on compilation,
unit tests, syntax checks, or static renderer assertions.

## Harness

The repeatable local harness is:

```bash
python3 scripts/dogfood-v09-user-cases.py
```

The script intentionally runs real commands rather than only calling Rust
helper functions:

- builds the local binary
- creates a temporary user project with a small Rust app under `/tmp` so the
  permission bridge Unix socket path stays below platform `SUN_LEN` limits
- runs `cr init --yes --preset team`
- inspects generated role inventory and composed host prompt
- verifies `.coreroom/priors.lock`
- switches the temporary project to gated `engine = "fake"` for dogfood only
- enters plain `cr` through a Unix PTY at `160x48`
- submits one normal prompt and one permission-triggering prompt through the
  TUI composer
- checks visible role status, streamed chunks, permission overlay text, and
  the absence of removed staged-room tokens
- verifies `.coreroom/messages.jsonl` gained `turn_dispatched`,
  `role_spoke`, and `permission_denied` events with real `turn_id` values
- enters `cr console --live-room` through a PTY and verifies the executable
  room opens
- enters `cr console` and `cr console --snapshot` through PTYs and exits with
  `q`
- regenerates README images

## Real Failures Caught During Harness Work

The harness caught non-syntax issues while being added:

- The earlier staged-room dogfood could show input text without proving any
  role engine executed. The fake engine now runs behind `EngineAdapter`, so the
  same runtime, TUI, permission bridge, and durable message path are exercised.
- Rendered status strings alone were too weak as proof. The harness now treats
  `.coreroom/messages.jsonl` turn events as the durable authority and uses
  visible strings only as tertiary checks.

These failures are exactly why this gate exists: realistic local paths expose
problems that pure syntax or helper-level tests do not.

## Not Verified

- Real authenticated Claude Code, Codex, or Gemini turns are still excluded
  from this gate because they require logged-in CLIs and can burn user/API
  quota.
- The fake engine proves CoreRoom's runtime contract: dispatch, streaming,
  permission prompt plumbing, TUI projection, and durable events. It does not
  prove vendor CLI authentication, remote model quality, or external tool
  behavior.
- The script uses Unix PTY APIs and is intended for macOS/Linux release
  dogfood. Windows terminal dogfood should be added as a separate platform gate
  before claiming native Windows console support.

## Release Rule

Any future release touching the console, TUI room, host-led control surface,
README visuals, setup/init path, or role prompt composition must run:

```bash
python3 scripts/dogfood-v09-user-cases.py
```

The result must be included in PR evidence. If this script fails, the release
is not ready even if `cargo test`, `clippy`, and CI pass.

## Rollback

If this dogfood gate proves too broad or flaky, revert the PR that updated this
document and `scripts/dogfood-v09-user-cases.py`. If the default TUI room must
be rolled back, restore plain `cr` to `cr start` routing while keeping
`cr console` as the read-only dashboard and reopening the runtime-parity
tracker.
